# M1.9.2.1.3 — Snapshot Reference Detachment Hotfix

## Origine del difetto

La prima esecuzione completa della suite M1.9.2.1.2 ha fatto emergere due failure equivalenti:

- `OpenRoundTest::testCatalogEditAndDeleteCannotRetroactivelyChangeARoundSnapshot`;
- `CatalogRoundGateVerifierTest::testGateScenarioPassesAndRollsBackEveryVerificationMutation`.

La cancellazione della coppia sorgente riusciva, ma `round_question.choice_pair_id` restava valorizzato invece di diventare `NULL`. L'identità storica `choice_pair_source_id_snapshot` restava corretta, ma il riferimento vivo diventava orfano.

## Causa

M1.9.2 affidava il distacco del riferimento vivo esclusivamente alla foreign key SQLite:

```text
choice_pair_id -> choice_pair(id) ON DELETE SET NULL
```

`PRAGMA foreign_keys` è connection-local. I normali ingressi HTTP/console applicano `SqliteRuntimeConfigurator`, ma i test di servizio `KernelTestCase` invocano direttamente i casi d'uso e non attraversano necessariamente gli eventi request/console. La suite ha quindi dimostrato che l'invariante di snapshot non deve dipendere soltanto da quel PRAGMA.

## Correzione

La migrazione `Version20260719000200` aggiunge il trigger SQLite:

```text
trg_choice_pair_detach_round_snapshot_reference
```

Prima della cancellazione di una coppia `REGULAR`, il trigger esegue atomicamente soltanto:

```text
round_question.choice_pair_id -> NULL
```

Non modifica mai:

- `choice_pair_source_id_snapshot`;
- testi/immagini snapshot;
- codice/categoria/tipo snapshot;
- step;
- `question_set_hash`.

Il trigger di immutabilità M1.9.2 consente già esclusivamente questa transizione del riferimento vivo e continua a respingere qualsiasi altra modifica.

`ON DELETE SET NULL` resta presente come difesa aggiuntiva quando `PRAGMA foreign_keys = ON`.

La migrazione ripara inoltre eventuali riferimenti vivi già orfani, impostandoli a `NULL` soltanto quando la coppia sorgente non esiste più; l'identità storica rimane nel campo snapshot immutabile.

## Gate

Eseguire:

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\scripts\verify-m1.9.2.1.3.ps1
```

oppure:

```bash
./scripts/verify-m1.9.2.1.3.sh
```

Il gate eredita integralmente M1.9.2.1.2 e quindi riesegue bootstrap, migrazioni, suite PHPUnit completa, gate transazionale catalogo/round e policy correttive.

## Criterio di accettazione

Le due verifiche che avevano evidenziato il difetto devono risultare verdi e, dopo la cancellazione di una coppia già snapshot-tata:

- `choice_pair_id IS NULL`;
- `choice_pair_source_id_snapshot` è invariato;
- lo snapshot storico è invariato;
- `question_set_hash` è invariato;
- il tamper diretto resta bloccato.
