# M1.9.2.1 — Runtime Compatibility & Monotonic Timing Hardening

## Stato

Milestone correttiva implementata sulla baseline M1.9.2 e **in attesa di validazione del gate completo**.

## Perché è necessaria

Un audit successivo alla preparazione di M1.9.2 ha individuato due incoerenze reali che devono essere chiuse prima di proseguire con M1.9.3.

### 1. Baseline PHP dichiarata incoerente con il lock

Il progetto dichiarava PHP 8.3+ in `composer.json`, README e preflight, ma il `composer.lock` conteneva dipendenze risolte che richiedono PHP 8.4. La CI eseguiva soltanto PHP 8.4, quindi la falsa compatibilità con PHP 8.3 non veniva esercitata.

Decisione progettuale: **PHP 8.4 è la baseline ufficiale di TwentyChoices**.

Correzioni:

- `composer.json` richiede `php >=8.4`;
- `config.platform.php` è fissato a `8.4.0`, così un `composer update` eseguito su runtime più nuovi non può selezionare accidentalmente dipendenze incompatibili con la baseline minima;
- la sezione `platform` di `composer.lock` è allineata a `>=8.4`, `platform-overrides.php` è `8.4.0` e il content hash è aggiornato;
- README, bootstrap PowerShell/Unix e preflight richiedono PHP 8.4+;
- `RuntimeBaselinePolicy` verifica automaticamente coerenza tra manifest, lock, README, bootstrap e CI;
- il bootstrap esegue `composer check-platform-reqs` subito dopo `composer install`;
- la CI esegue lo stesso controllo;
- PHP 8.3 viene considerato esplicitamente non supportato.

Questo rafforza retroattivamente il gate M1.9.1: non basta più che il PHP locale superi un controllo numerico, la piattaforma reale deve soddisfare anche le dipendenze installate.

### 2. Countdown browser dipendente dall'orologio civile del client

Il template esponeva `shownAt` e `availableAt` come epoch server e `public/play.js` li confrontava con `Date.now()`.

Con un clock client sfasato:

- client indietro: attesa visuale più lunga del necessario;
- client avanti: pulsante abilitato troppo presto, seguito dal corretto rifiuto server `ChoiceTooEarly` e da una UX degradata.

La sicurezza server-side non era bypassabile, ma il comportamento client non era robusto.

Correzione:

- `OpenPlayStep` calcola sul server `waitRemainingMilliseconds` ed `elapsedSinceShownMilliseconds` al momento della costruzione della schermata;
- il template espone esclusivamente queste durate relative;
- gli epoch assoluti non sono più inseriti nel DOM;
- `play.js` usa `performance.now()` come clock monotono locale;
- `Date.now()` non è più usato nel countdown o nella telemetria del tempo trascorso;
- il controllo autorevole dei due secondi resta invariato e continua a essere esclusivamente server-side.

Il piccolo tempo di trasporto/render successivo al calcolo server può soltanto rendere il countdown client leggermente conservativo; non può anticipare l'accettazione server.

## Test e gate automatici

Nuovi controlli:

- `tools/runtime-baseline-check.php`;
- `tools/client-timing-check.php`;
- test che rifiuta PHP 8.3 e accetta PHP 8.4+;
- test di coerenza delle dichiarazioni runtime distribuite;
- test statico anti-regressione che vieta `Date.now()` nel timer e gli epoch assoluti nel DOM;
- test applicativi che verificano 2000/0 ms alla prima apertura, 1000/1000 ms dopo un secondo e clamp a 0 ms di attesa dopo la scadenza.

Comando Windows:

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\scripts\verify-m1.9.2.1.ps1
```

Comando Linux/macOS:

```bash
./scripts/verify-m1.9.2.1.sh
```

Totale suite predisposta dopo la correzione: **95 metodi PHPUnit / 97 casi effettivi**, oltre alle **20 verifiche indipendenti** di `tools/domain-tests.php`.

Il gate esegue:

1. package audit;
2. runtime baseline policy;
3. client timing policy;
4. bootstrap completo e suite regressiva;
5. `composer check-platform-reqs` tramite bootstrap;
6. gate transazionale M1.9.2 catalogo/round;
7. riesecuzione finale delle due policy correttive.

## Osservazioni dell'audit rinviate ai gate già previsti

Le altre osservazioni non richiedono patch anticipata in questa milestone:

- concorrenza del sequence number dell'audit: da stressare in M1.9.8 e riconciliare in M1.9.11;
- cleanup del rate limiter: da misurare in M1.9.12/M1.9.14 prima di ottimizzare;
- stato `CREATED` non usato nel normale flusso: debito di modello, non bug funzionale corrente;
- stato parziale del reveal: già impedito dai trigger SQLite; M1.9.10/M1.9.13 ne verificheranno l'invariante;
- riconnessioni DBAL in runtime long-running: fuori dall'architettura corrente, da considerare nei limiti operativi M1.9.14.

## Gate

**M1.9.2.1 è superata soltanto quando PHP 8.4+ completa l'intero script di verifica senza errori, Composer conferma i platform requirements e i controlli anti-regressione del timer risultano verdi. La hotfix M1.9.2.1.1 rende il gate rieseguibile anche su una working copy già inizializzata.**

Dopo la validazione, M1.9.2 e la correzione M1.9.2.1 formano insieme la nuova baseline per M1.9.3.

## Hotfix M1.9.2.1.1

Il gate è stato reso rieseguibile su working copy già inizializzate mediante `release-manifest.json` e `tools/release-manifest-check.php`. Il package audit rigoroso resta separato come controllo di release pre-confezionamento.
