# M1.9.1 — Environment & Database Verification

## Stato

**VALIDATA dall'utente.** La verifica completa su estrazione pulita è risultata tutta verde: bootstrap ripetibile, controlli installazione e suite completa superati. La baseline risultante è TwentyChoices M1.9.1.

> Nota correttiva M1.9.2.1: la baseline runtime originariamente dichiarata come PHP 8.3+ è stata successivamente resa coerente con il lock effettivo e fissata a **PHP 8.4+**. Il gate corrente M1.9.1 eredita quindi il preflight e il controllo platform aggiornati.

M1.9.1 non introduce nuove regole di gioco né modifica settlement, commitment, catalogo o schema applicativo. Rafforza esclusivamente bootstrap, diagnostica d'installazione, ripetibilità dei test e prove del pacchetto distribuito.

## Obiettivo

Dimostrare che TwentyChoices può essere estratto in una cartella vuota e inizializzato senza dipendenze implicite dal computer di sviluppo, con:

- runtime PHP verificato prima di Composer;
- segreto locale generato fuori dal pacchetto;
- SQLite realmente operativo tramite il PHP CLI in uso;
- database `dev` e `test` separati;
- migrazioni complete da zero e idempotenti quando già applicate;
- seed iniziale coerente;
- PRAGMA runtime verificabili;
- database test ricreato deterministicamente prima di ogni suite;
- bootstrap ripetibile due volte;
- procedura equivalente su Windows PowerShell e Unix shell.

## Rafforzamenti introdotti

### 1. Preflight condiviso e testabile

`tools/bootstrap-preflight.php` usa `tools/BootstrapPreflight.php` e verifica prima dell'avvio di Symfony:

- PHP >= 8.4;
- estensioni `ctype`, `iconv`, `PDO`, `pdo_sqlite`;
- presenza effettiva del driver `sqlite` in `PDO::getAvailableDrivers()`;
- apertura reale di una connessione `sqlite::memory:`;
- backend crittografico autenticato Sodium secretbox oppure OpenSSL AES-256-GCM;
- creazione e scrittura della directory `var`.

`scripts/bootstrap.ps1` e `scripts/bootstrap.sh` usano ora lo stesso verificatore, eliminando divergenze tra i due percorsi di installazione.

### 2. Audit del pacchetto pulito

`tools/package-audit.php` usa `tools/PackageAudit.php` per rifiutare un pacchetto che contenga materiale runtime o locale, tra cui:

- `.env.local`, `.env.local.php` e `.env.*.local`;
- `vendor/`;
- database SQLite e sidecar WAL/SHM/journal;
- file runtime sotto `var/` diversi dall'eventuale `.gitignore`;
- `phpunit.xml` locale e cache PHPUnit;
- un `APP_SECRET` reale inserito per errore nel `.env` distribuito.

Il placeholder storico `change-this-development-secret` nel `.env` non è considerato un segreto e viene comunque sovrascritto dalla `.env.local` generata dal bootstrap.

### 3. Verifica persistente dell'installazione

Nuovo comando:

```bash
php bin/console app:installation:verify
```

Controlla sul database realmente aperto:

- percorso SQLite coerente con l'ambiente (`var/data.db` oppure `var/test.db`);
- separazione configurata tra database sviluppo e test;
- `PRAGMA quick_check`;
- foreign keys abilitate;
- `busy_timeout >= 5000`;
- `synchronous` almeno `FULL`;
- WAL fuori dall'ambiente test;
- corrispondenza esatta tra migrazioni presenti in `migrations/` e migrazioni registrate come applicate;
- almeno 19 coppie regolari attive e una sola porta finale di sistema attiva;
- `APP_SECRET` locale non-placeholder e sufficientemente lungo fuori dai test.

Il bootstrap esegue questo comando subito dopo le migrazioni e prima di `app:system:check`.

### 4. Reset deterministico del database test

La cancellazione di:

```text
var/test.db
var/test.db-wal
var/test.db-shm
var/test.db-journal
```

è stata estratta in `tools/TestDatabaseReset.php` ed è ora testabile separatamente. `bin/phpunit` continua a ricreare il database e ad applicare tutte le migrazioni prima di avviare PHPUnit.

### 5. Verifica automatica del gate

Sono disponibili due script equivalenti:

Windows:

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\scripts\verify-m1.9.1.ps1
```

Linux/macOS:

```bash
./scripts/verify-m1.9.1.sh
```

Da una cartella appena estratta eseguono nell'ordine:

1. audit del pacchetto pulito;
2. primo bootstrap completo;
3. secondo bootstrap completo sullo stesso stato, per verificare l'idempotenza;
4. verifica finale con `app:installation:verify`.

La CI esegue inoltre due volte consecutive `doctrine:migrations:migrate --env=test` prima della verifica d'installazione, rendendo esplicito il controllo di idempotenza delle migrazioni già applicate.

## Test automatici aggiunti

- `BootstrapPreflightTest`
  - verifica che il runtime usato dalla suite soddisfi tutti i prerequisiti obbligatori;
- `PackageAuditTest`
  - accetta un pacchetto sorgente pulito;
  - rifiuta segreti locali, database, vendor e artefatti runtime;
- `TestDatabaseResetTest`
  - verifica la rimozione deterministica di DB/WAL/SHM/journal anche con due reset consecutivi;
- `InstallationVerifierTest`
  - verifica su database test reale percorso, migrazioni, seed e PRAGMA critici.

La suite passa da **79 a 84 metodi PHPUnit**, pari a **86 casi effettivi** considerando il data provider di `WinningPathTest`. Il runner indipendente resta a **20 verifiche**.

## Risultati osservati nell'ambiente di preparazione

Con PHP 8.4.16:

- `php -l` sui file nuovi/modificati PHP: **OK**;
- `php tools/package-audit.php`: **OK**;
- `php tools/domain-tests.php`: **20/20 passati**;
- il nuovo preflight segnala correttamente come errore l'assenza di `pdo_sqlite` nel PHP CLI dell'ambiente di preparazione.

La suite Symfony/PHPUnit completa **non è stata dichiarata eseguita** in questo ambiente perché il runtime disponibile non espone `pdo_sqlite` e non dispone di Composer. Questa è precisamente una delle condizioni che M1.9.1 deve intercettare in modo fail-fast.

## Bug/gap individuati e corretti

1. **Controlli bootstrap duplicati tra PowerShell e shell**  
   Rischio: divergenza futura tra piattaforme.  
   Correzione: preflight PHP unico condiviso.

2. **Reset del database test incorporato direttamente nel wrapper PHPUnit**  
   Rischio: comportamento importante ma non testabile isolatamente.  
   Correzione: estratto `TestDatabaseReset` con test dedicato.

3. **Assenza di una prova automatica che il pacchetto distribuito sia privo di runtime/segreti locali**  
   Correzione: `PackageAudit` e relativo test.

4. **Assenza di una verifica unica post-migrazione per path DB, migrazioni, seed e PRAGMA**  
   Correzione: `app:installation:verify` e test d'integrazione.

Non sono state individuate né modificate regole funzionali del gioco.

## Checklist manuale di accettazione

La prova va eseguita su una **nuova estrazione dello ZIP M1.9.1**, non su una cartella già inizializzata.

### Windows PowerShell

```powershell
powershell.exe -ExecutionPolicy Bypass -File .\scripts\verify-m1.9.1.ps1
```

### Esito richiesto

Verificare che:

- `Package audit OK` compaia prima di qualunque installazione;
- il preflight mostri tutti i controlli `[OK]`;
- al primo passaggio venga generato `.env.local`;
- `var/data.db` venga creato e tutte le migrazioni applicate;
- `app:installation:verify` termini senza errori;
- `app:system:check` non segnali errori critici;
- runner indipendente e PHPUnit siano verdi;
- il secondo bootstrap non crei duplicati, non corrompa il catalogo e riporti le migrazioni già allineate;
- PHPUnit ricrei nuovamente `var/test.db` e termini verde;
- la verifica finale d'installazione sia verde.

Controllo opzionale esplicito dei due database:

```powershell
Get-Item .\var\data.db
Get-Item .\var\test.db
```

Devono essere due file distinti.
